# Carapace Man Page Documentation

Reference for [carapace-man](https://github.com/carapace-sh/carapace-man) вЂ” providing inline documentation for completion values via YAML man pages.

## Overview

carapace-man provides contextual documentation for completion values. When a user selects a completion candidate, the UID of that action maps to a YAML file containing markdown descriptions. These descriptions are rendered inline (e.g. in the shell's completion menu) to help users understand what each completion value means.

Every UID is built the same way: `scheme://host/value`, where the value is the completion candidate itself. What differs is whether that value is a **known concept** or a **live entity**:

- `jj://revset-function/parents` вЂ” `parents` is a built-in function; its meaning is fixed and universal
- `jj://bookmark/main` вЂ” `main` is a bookmark; its meaning depends on which repo you're in

The YAML files are organized by scheme and host, and each value becomes a key in the YAML map. For known concepts, you can write all the keys upfront. For live entities, the keys are only known at runtime.

### cmd/ вЂ” Command Specs

The `cmd/` directory contains split YAML command specs (one file per subcommand). These are generated by `carapace-man split` from a full spec. See [spec.md](spec.md) for the YAML spec format.

### Non-cmd/ вЂ” Domain Documentation

All other top-level directories are **scheme** directories. Each scheme contains **host** subdirectories, and each host directory contains a single YAML file named `<host>.yaml`.

The UID `scheme://host/path` maps to:
- **File**: `man/<scheme>/<host>/<host>.yaml`
- **Key**: `path` (the path component of the UID, without leading `/`)

## YAML Format

Each YAML file is a map of **key в†’ markdown description**:

```yaml
key1: |
  Markdown description for key1.
  Can span multiple lines and include `code`, **bold**, and [links](scheme://host/path).
key2: |
  Markdown description for key2.
```

- Keys are the path component of the UID (e.g. for `jj://revset-function/parents`, the key is `parents`)
- Values are markdown strings rendered with [glamour](https://github.com/charmbracelet/glamour)
- Use the `|` YAML block scalar for multi-line descriptions
- Cross-references use UID-style links: `[text](scheme://host/path)`

### `./` Path Indirection

When a YAML value starts with `./`, it is treated as a relative path to another file whose content becomes the description. This is useful for large descriptions that don't fit inline:

```yaml
large_topic: ./detailed.md
```

The path is resolved relative to `man/<scheme>/<host>/`. The referenced file's entire content is returned as-is (not parsed as YAML).

## Known Concepts

Some UIDs refer to things whose meaning is fixed вЂ” a revset function, an HTTP method, a fileset operator. The documentation for these is the same everywhere and can be written once, covering every possible value.

### Examples

File: `man/jj/revset-function/revset-function.yaml`

```yaml
parents: |
  `parents(x, [depth])`: `parents(x)` is the same as `x-`.
  `parents(x, depth)` returns the parents of `x` at the given `depth`.
all: |
  `all()`: All visible commits and ancestors of commits explicitly mentioned.
```

File: `man/jj/fileset-operator/fileset-operator.yaml`

```yaml
"&": |
  `x & y`: Intersection вЂ” matches files in both `x` and `y`.
"|": |
  `x | y`: Union вЂ” matches files in `x` or `y`.
"~": |
  `x ~ y`: Difference вЂ” matches files in `x` but not in `y`.
```

File: `man/http/method/method.yaml`

```yaml
GET: |
  The [GET](http://method/GET) HTTP method requests a representation of the specified resource.
POST: |
  The [POST](http://method/POST) HTTP method sends data to the server.
```

## Live Entities

Other UIDs refer to things whose meaning depends on context вЂ” a bookmark name, a commit ID, a file path. The same key resolves to different content in different repos or on different machines:

- `jj://bookmark/main` в†’ different commit in each repo
- `jj://commit/abc123` в†’ different change depending on the repo
- `file:///etc/hosts` в†’ different contents on each machine

The YAML file for a live-entity host contains a **conceptual entry** вЂ” a representative key that documents what kind of thing this category is. The actual per-value descriptions come from the live context at runtime.

### Example

File: `man/jj/bookmark/bookmark.yaml`

```yaml
local: |
  A local bookmark in the jj repository. Bookmarks are jj's equivalent of Git branches.

  Local bookmarks point to a single commit. Use `jj bookmark list` to see all local bookmarks.
```

The key `local` is a placeholder вЂ” at runtime the actual keys are the bookmark names from the current repository (`main`, `develop`, etc.). Since `Describe()` does an exact key lookup, a runtime value like `main` won't match `local`. The completion action's own `ActionValuesDescribed` descriptions (bookmark name в†’ commit message) serve as the primary per-value documentation in this case.

### Populating Live-Entity Keys

To make `Describe()` resolve for live-entity UIDs, the YAML needs to contain the actual runtime values as keys. Approaches:

1. **Pre-populate common values** вЂ” add well-known entries (e.g. `main`, `trunk`) alongside the conceptual entry
2. **Sync from live context** вЂ” a generation step queries the system and writes actual keys with their current descriptions (e.g. `jj bookmark list` в†’ populate `bookmark.yaml` with nameв†’message entries)

The `./` path indirection can also point to a generated markdown file for a specific value.

## UID Resolution

The `Describe()` function resolves a UID to its description:

1. **`cmd://` scheme**: Loads the split command spec YAML and returns the command/flag documentation
2. **`man://` scheme**: Looks up a system man page by name and section
3. **Other schemes**: Loads `man/<scheme>/<host>/<host>.yaml` and looks up the key matching the UID's path

```go
// UID: jj://revset-function/parents
// File: man/jj/revset-function/revset-function.yaml
// Key:  "parents" (UID.Path without leading "/")
```

## Adding Documentation for a New Scheme

1. Create the directory: `man/<scheme>/<host>/`
2. Create the YAML file: `man/<scheme>/<host>/<host>.yaml`
3. For **known concepts**: add entries for each possible value
4. For **live entities**: add a conceptual entry documenting the value type; optionally pre-populate common values or add a sync step
5. Set UIDs on the corresponding actions using `.Uid()` or `.UidF()`

## carapace-man CLI

The `carapace-man` tool provides subcommands:

| Command | Description |
|---------|-------------|
| `split <spec>` | Split a full YAML spec into per-subcommand files in `cmd/` |
| `sync` | Sync man pages to the config directory |
