## Global Flags

These flags are available on all kubeadm commands (defined on the root command):

| Flag | Description |
|------|-------------|
| `--rootfs` | Path to the 'real' host root filesystem. kubeadm will chroot into the provided path. Useful for testing or operating on a mounted disk. |
| `--v` | Number for the log level verbosity. Higher = more verbose (e.g. `--v=5`). |
| `--vmodule` | Comma-separated list of `pattern=N` settings for file-filtered logging. |

### `--rootfs`

Causes kubeadm to chroot into the specified path before running. This is useful when:
- Repairing a node from a live USB/recovery environment
- Operating on a mounted disk image
- Testing kubeadm against a fake filesystem

```bash
kubeadm init --rootfs /mnt/host
```

## `--kubeconfig`

Used by commands that talk to an existing cluster (`token`, `config`, `upgrade`, `reset`, `certs`).

| Default | Description |
|---------|-------------|
| `/etc/kubernetes/admin.conf` | The kubeconfig file to use when talking to the cluster. |

If not set, kubeadm searches standard locations for an existing kubeconfig file. The admin kubeconfig generated by `init` is at `/etc/kubernetes/admin.conf`.

```bash
kubeadm upgrade apply v1.30.0 --kubeconfig /custom/path/admin.conf
```

## `--dry-run`

Available on `init`, `join`, `reset`, `upgrade apply`, `upgrade node`, and `token create`.

Outputs what would be done without applying any changes. Useful for:
- Validating a configuration before committing
- Previewing the upgrade diff
- Testing preflight checks

```bash
kubeadm init --dry-run
kubeadm upgrade apply v1.30.0 --dry-run
```

## `--ignore-preflight-errors`

Available on all commands that run preflight checks (`init`, `join`, `reset`, `upgrade apply`, `upgrade node`).

A comma-separated list of check names whose errors will be shown as warnings instead of causing a failure. The special value `all` ignores errors from all checks.

```bash
kubeadm init --ignore-preflight-errors=Swap,IsPrivilegedUser
kubeadm init --ignore-preflight-errors=all
```

### Preflight Checks

| Check | Description |
|-------|-------------|
| `ContainerRuntime` | Container runtime (Docker/containerd) is installed and running |
| `DirAvailable` | Required directories are available |
| `ExternalEtcdVersion` | External etcd version is compatible |
| `FileAvailable` | Required files are available |
| `FileContent` | File contents are valid |
| `FileExisting` | Required files exist |
| `Firewalld` | Firewalld is not active (can interfere with ports) |
| `HTTPProxy` | HTTP proxy settings |
| `HTTPProxyCIDR` | HTTP proxy CIDR settings |
| `Hostname` | Hostname is valid |
| `ImagePull` | Can pull required images |
| `InPath` | Required binaries are in PATH |
| `IsPrivilegedUser` | Running as root/privileged user |
| `KubeletVersion` | Kubelet version is compatible |
| `KubernetesVersion` | Kubernetes version is valid |
| `Mem` | Sufficient memory available |
| `NumCPU` | Sufficient CPU cores |
| `PortOpen` | Required ports are available |
| `Service` | Required services are active |
| `Swap` | Swap is disabled (Kubernetes requires no swap) |
| `SystemVerification` | System meets requirements |
| `all` | Ignore all preflight errors (special value) |

**Warning**: Using `all` or ignoring critical checks like `IsPrivilegedUser` can lead to a broken cluster. Use selectively and understand each check you're suppressing.

## `--cri-socket`

Available on `init`, `join`, `reset`, and `config images pull`.

Specifies the path to the CRI (Container Runtime Interface) socket. If empty, kubeadm auto-detects the socket. Use this only if:
- You have more than one CRI installed (e.g. both containerd and CRI-O)
- You have a non-standard CRI socket path

```bash
kubeadm init --cri-socket unix:///var/run/containerd/containerd.sock
kubeadm init --cri-socket unix:///var/run/crio/crio.sock
```

## `--patches`

Available on `init`, `join`, `upgrade apply`, and `upgrade node`.

Path to a directory containing patch files for static Pod manifests and component configurations. This allows customizing control plane components without modifying the kubeadm config.

### Patch File Naming

```
target[suffix][+patchtype].extension
```

| Component | Target name |
|-----------|-------------|
| API Server | `kube-apiserver` |
| Controller Manager | `kube-controller-manager` |
| Scheduler | `kube-scheduler` |
| etcd | `etcd` |
| Kubelet config | `kubeletconfiguration` |
| CoreDNS deployment | `corednsdeployment` |

| Patch type | Description |
|------------|-------------|
| `strategic` | Strategic merge (default) |
| `merge` | JSON merge patch |
| `json` | JSON patch (RFC 6902) |

| Extension | Format |
|-----------|--------|
| `.yaml` | YAML |
| `.json` | JSON |

### Examples

```bash
# Apply patches from a directory
kubeadm init --patches /path/to/patches
```

Patch file examples:
- `kube-apiserver+merge.yaml` â€” JSON merge patch for the API server
- `kube-apiserver0+strategic.yaml` â€” Strategic merge patch, applied first (suffix `0`)
- `etcd.json` â€” JSON patch for etcd

The optional `suffix` is an alphanumeric string that determines application order when multiple patches target the same component.

### How Patches Work

Patches are applied to the generated static Pod manifests before they are written to `/etc/kubernetes/manifests/`. This lets you add extra args, volumes, environment variables, or other customizations to control plane components.

## `--config`

Available on most commands. Passes a YAML configuration file instead of using individual flags. See [config.md](config.md) for the full configuration API.

```bash
kubeadm init --config kubeadm-init.yaml
kubeadm join 10.0.0.1:6443 --config kubeadm-join.yaml
```

## `--skip-phases`

Available on `init`, `join`, `reset`, `upgrade apply`, and `upgrade node`. See [phases.md](phases.md) for details.

## `--feature-gates`

Available on `init`, `join`, and `config images` commands. See [concepts.md](concepts.md) for the feature gate lifecycle and current gates.

```bash
kubeadm init --feature-gates NodeLocalCRISocket=true
```
